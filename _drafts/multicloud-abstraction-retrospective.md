## TL;DR

- 사례 하나(AWS)만 보고 미래(Azure)를 **상상으로** 추상화했다.
- **행위 다형성**과 **테이블 상속**을 하나로 엉겨붙인 게 핵심 실수였다.
- 근본 원인은 JPA `JOINED`가 아니라 **성급한 추상화 + DB 설계 미숙**이었다.
- 지금이라면 DB·행위·API 세 계층에 같은 규칙을 적용한다:
  **검증된 공통만 위로 올리고, 실제 이질성은 따라 분리한다. 상상으로 미리 통합하지 않는다.**

## 1. 배경 — 무엇을 만들었나

AWS/Azure 클라우드 모니터링 제품을 만들었다. 내가 맡은 범위는 대략:

- **DB 설계** — cloud account / instance(resource) / metric / collector 등 도메인 모델링
- **메트릭 관리(collector)** — 어떤 지표를 수집할지 말지 정의. (모든 지표를 수집하면 비용이
  드니까) 리전별·서비스별(EC2 등)·메트릭별·집계별(min/max/sum/avg 등) **수집 여부와 수집
  주기**를 설정하는 화면
- **모니터링 화면** — 수집한 메트릭을 가져와 보여주는 화면
- **account 자동 갱신** — 일정 주기로 비동기로 계정을 갱신해 인스턴스(리소스)를 최신화

클라우드 도메인 지식 자체(계정 인증 방식, 리전/서비스 구분, 지표 종류 등)는 수집 담당자가
공부해서 내게 넘겨줬고, 나는 그 정보를 기반으로 **"나중에 클라우드 종류가 추가돼도 돌아가는
구조"** 를 짜는 데 집중했다.

## 2. 내가 한 설계와 그렇게 한 이유

모든 클라우드 대상에 대해 추상 엔티티를 만들었다:

- `cloud_account` ← `aws_account`, (나중에) `azure_account`
- `cloud_instance`, `cloud_metric`, `cloud_collector` … 도 같은 식
- 집계 구조도 계층으로: `cloud_collector` → `cloud_collector_metric`(대표 지표) →
  `cloud_metric`(sum 등 각각) → `aws_data`(실제 메트릭 값)

왜 이렇게 했나:

- **인터페이스의 힘 / 클린코드**의 영향을 크게 받았다.
- 클라우드(AWS/Azure/…)를 하나하나 더하는 게 아니라, **`enum`(AWS/AZURE)에 값 하나만
  추가하면 뚝딱 되는** 구조를 원했다.

## 3. Azure가 오자 무너진 지점

AWS만 있을 땐 문제가 안 보였다. `cloud_account`를 전부 조회한 뒤 그냥 `(AwsAccount)`로
캐스팅해도 됐으니까. 그런데 `azure_account`가 생기니 그런 것들이 다 깨졌다.

지금 와서 이름을 붙이면:

- **`(AwsAccount)` 캐스팅이 됐던 건 다형성이 아니었다.** 단일 타입이 추상 클래스 옷만
  입고 있던 것. 타입이 하나뿐이라 안 터졌을 뿐, 진짜 다형적 구조가 아니었다.
- **행위 다형성과 테이블 상속을 엉겨붙였다.** "provider 상관없이 동작하게" 하고 싶은 건
  *코드의 행위* 문제인데, 그걸 *테이블 상속*으로 풀려고 했다. 이 둘은 별개 결정이다.
- **공통을 조사 없이 상상으로 못박았다.** AWS/Azure의 실제 차이(예: 계정 인증 방식,
  region/service vs subscription/resourceGroup 같은 계층·식별 체계)를 제대로 조사하지도
  않고 공통 필드를 정했다. <!-- TODO: 실제 차이점 구체적으로 채우기 -->
- **과잉 계층.** collector→collector_metric→metric→aws_data 4층은 실제 불변식이 아니라
  "개념적으로 깔끔해 보여서" 만든 정규화였다.

솔직히 `JOINED` 전략 자체보다도 **DB 설계 감각이 없던 사람이 추상화의 힘에 빠져** 복잡하게
만들어버린 게 근본 원인이었다. 어디서 뭐가 문제인지도 모를 만큼 얽혔다.

## 4. 지금이라면 — 세 계층, 같은 규칙

### (1) 저장(DB): 클라우드마다 테이블을 따로 둔다

- 공통 필드는 **`@MappedSuperclass`** 추상 클래스로 뽑는다.
  **`@Entity` + `@Inheritance(JOINED/SINGLE_TABLE)`가 아니다.**
- `@MappedSuperclass`는 테이블이 없고, 필드만 각 구체 테이블(`aws_account`,
  `azure_account`)로 복사된다. → **폴리모픽 조회도, 크로스-provider 캐스팅도 사라진다.**
  내가 겪은 고통의 원인이 통째로 제거된다.
- 게다가 "공통인 줄 알았는데 아니네 → AWS로 내리기"가 테이블 계층을 안 건드려서
  리팩터링이 싸다.
- **베이스엔 진짜 보편적인 것만** 둔다(id, provider enum, 감사필드, 이름 정도). region이나
  credential 같은 도메인 필드는 **2개 이상 provider가 실제로 공통임을 확인해줄 때** 올린다.

### (2) 행위(코드): 공통 흐름은 인터페이스/SPI로

- provider별 구현체를 만들고 registry에 등록한다.
- **"enum 하나 추가하면 뚝딱"의 꿈이 유일하게 진짜 성립하는 곳이 여기다.**
  행위는 닫고(OCP), 스키마·DTO는 닫지 않는다.

### (3) API: 모양이 갈리면 분리한다

- AWS/Azure의 요청·응답 DTO가 실제로 다르면 분리한다(억지 union DTO 금지).
- 단, 완전히 동일한 오퍼레이션까지 반사적으로 쪼개는 것도 과잉이다. 기준은 DB와 같다 —
  **실제 이질성을 따라간다.**

## 5. 관통 원칙

> **DB·행위·API 세 계층 모두 같은 규칙이다.**
> 검증된 공통만 위로 올리고, 실제 이질성은 따라 분리한다. 상상으로 미리 통합하지 않는다.
> 행위만 인터페이스로 닫고(SPI), 저장·전송 모양은 provider를 따라간다.
{: .prompt-info }

그리고 **Rule of Three**: AWS 하나 구체적으로 → Azure 하나 구체적으로 → *그제야* 실제
공통을 추출한다. 2개 이상 실물에서 발견한 추상은 살아남고, 1개 + 상상으로 만든 추상은
나와 싸운다.
